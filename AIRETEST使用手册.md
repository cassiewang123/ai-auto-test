## 1. 先记住三件事

### 1.1 启动平台只使用这一条命令

在项目根目录执行：

```powershell
Set-Location E:\xlwang\AIRETEST
.\scripts\start-local.ps1
```

不要同时手工启动多个后端、Vite 或 Uvicorn 进程。启动脚本会自动完成：

1. 检查 `.env` 和本地依赖。
2. 执行 SQLite 数据库迁移。
3. 启动后端。
4. 启动前端。
5. 等待前后端健康检查通过。

### 1.2 默认是轻量单机模式

本地默认使用：

```text
数据库：SQLite
任务执行：当前后端进程内的本地后台线程
前端：http://127.0.0.1:5173
后端：http://127.0.0.1:8001
API 文档：http://127.0.0.1:8001/docs
```

不需要安装 Docker、Oracle、Redis 或 Celery Worker。C-Lodop 使用的
`8000` 端口不属于 AIRETEST，不要为了启动平台去停止它。

### 1.3 顶部的“项目”和“环境”不是装饰

登录后，页面顶部会显示两个选择器：

- **项目**：决定当前查看和编辑哪一组接口、用例、报告等资源。
- **环境**：提供 Base URL、环境变量、Cookie 和可选的数据库配置。

没有项目时，很多列表会为空；没有环境时，相对 URL、环境变量和数据库断言
可能无法按预期执行。第一次使用时，先创建项目，再创建环境。

## 2. 第一次启动

### 2.1 检查工具版本

在 PowerShell 执行：

```powershell
python --version
node --version
npm --version
```

建议：

| 工具 | 建议版本 |
|---|---|
| Python | 3.13 |
| Node.js | 20.19+，或 22.12+ |
| PowerShell | Windows PowerShell 5.1 或 PowerShell 7 |

### 2.2 准备 `.env`

项目根目录应该已有 `.env`。如果没有，执行：

```powershell
Set-Location E:\xlwang\AIRETEST
Copy-Item .env.example .env
```

轻量模式至少应保持以下配置：

```dotenv
DATABASE_URL=sqlite:///./airetest-lite.db
TASK_DISPATCH_MODE=local
TASK_FALLBACK_MODE=disabled
AUTO_CREATE_SCHEMA=false
ARTIFACT_ROOT=.uploads
VITE_AIRETEST_MODE=lite
```

如果你是从全新目录开始，还需要首次安装依赖：

```powershell
Set-Location E:\xlwang\AIRETEST
python -m pip install -r backend\requirements.txt
python -m pip install .\test-engine
Set-Location frontend
npm ci
Set-Location ..
```

通常情况下，`start-local.ps1` 会自动检查依赖；只有依赖不完整时才需要手工
执行上面的安装命令。

### 2.3 启动

```powershell
Set-Location E:\xlwang\AIRETEST
.\scripts\start-local.ps1
```

看到“轻量单机模式已启动”后，打开浏览器访问：

```text
http://127.0.0.1:5173
```

### 2.4 查看运行状态

另开一个 PowerShell 窗口执行：

```powershell
Set-Location E:\xlwang\AIRETEST
.\scripts\status-local.ps1
```

重点确认：

- `Backend process` 正在运行。
- `Backend ready` 为 `True`。
- `Frontend process` 正在运行。
- `Frontend ready` 为 `True`。

### 2.5 停止平台

```powershell
Set-Location E:\xlwang\AIRETEST
.\scripts\stop-local.ps1
```

停止不会删除 SQLite 数据、测试产物、日志或初始管理员信息。下次仍然使用
`.\scripts\start-local.ps1` 启动即可。

### 2.6 PowerShell 提示禁止执行脚本

只对当前 PowerShell 窗口临时放开：

```powershell
Set-ExecutionPolicy -Scope Process Bypass
```

然后重新执行启动命令。不要为了这个问题修改整台电脑的永久执行策略。

## 3. 登录平台

### 3.1 已有数据库

本机初始管理员信息保存在：

```text
E:\xlwang\AIRETEST\.runtime\initial-admin.txt
```

请只在本人电脑上查看这个文件。不要把里面的密码写入手册、截图、Git 提交
或发送给其他人。

通常用户名是 `admin`，但应以该文件或你当前设置的密码为准。登录页输入：

1. 用户名。
2. 密码。
3. 点击“登录”。

### 3.2 全新数据库

如果是全新的 SQLite 数据库，登录页可以点击“注册首个管理员”：

1. 输入用户名，至少 3 个字符。
2. 输入邮箱。
3. 输入密码，至少 6 个字符。
4. 点击“注册”。
5. 回到登录页登录。

第一个注册账号会自动成为管理员。

### 3.3 登录成功后的页面

登录成功后会进入“仪表盘”。右上角可以看到当前用户，点击用户区域可以
退出登录。登录状态失效时，重新登录即可。

## 4. 先理解页面布局

### 4.1 左侧菜单

默认的 `lite` 菜单包含个人单机使用最常用的功能：

| 菜单 | 用途 |
|---|---|
| 仪表盘 | 查看平台总体统计和快捷入口 |
| 任务中心 | 查看 API、UI、性能任务的排队、执行和结果 |
| API 测试 | 接口定义、接口调试、用例、计划、数据驱动、导入、Mock |
| UI 测试 | UI 用例、测试套件、步骤库、元素对象库、调用记录、日志 |
| 性能测试 | 压测场景、性能报告、实时仪表盘 |
| 测试报告 | 查看测试运行结果并导出报告 |
| 覆盖率看板 | 查看接口覆盖率 |
| 定时任务 | 按计划配置定时执行 |
| 环境管理 | 管理 Base URL、变量、Cookie 和数据库连接 |
| 变量管理 | 管理全局变量和项目工作空间变量 |
| 项目管理 | 创建和管理项目 |
| 历史记录 | 查看快速调试产生的调用历史 |
| AI 助手 | 生成测试点、用例、断言或分析失败 |

“知识工程”和“系统管理”等治理功能默认隐藏。确实需要查看时，可在项目
根目录执行：

```powershell
.\scripts\start-local.ps1 -FrontendMode full
```

切回轻量菜单：

```powershell
.\scripts\start-local.ps1 -FrontendMode lite
```

### 4.2 顶部工作区选择器

顶部的“项目”和“环境”选择器会影响当前页面的数据范围。平台会记住你上次
选择的项目和环境。

建议：

1. 创建项目后，立即在顶部选择该项目。
2. 创建环境后，立即在顶部选择该环境。
3. 页面显示“暂无数据”时，先确认顶部项目和环境是否正确。
4. UI 套件、性能场景等页面还可能有自己的项目筛选器，也要检查。

### 4.3 页面中的常见按钮

- **新建**：创建资源。
- **编辑**：修改资源。
- **执行/运行**：启动测试。
- **刷新**：重新从后端读取数据。
- **详情**：查看执行步骤、请求响应或统计。
- **删除**：删除资源，执行前请确认。
- **清空**：通常不可恢复，只用于清理历史记录。

## 5. 第一次使用的推荐顺序

第一次不要一开始就配置性能测试、AI 或数据库断言。建议按下面顺序走完
最小闭环：

1. 启动平台并登录。
2. 在“项目管理”创建一个项目。
3. 在“环境管理”创建一个环境。
4. 在顶部选择刚创建的项目和环境。
5. 在“接口调试”发送一个简单 GET 请求。
6. 点击“保存到接口列表”。
7. 在“接口定义”或“用例管理”确认保存成功。
8. 创建一个测试计划并加入用例。
9. 执行单个用例，或按需配置定时任务。
10. 在“任务中心”“测试报告”“历史记录”查看结果。

如果只是临时验证一个接口，做到第 5 步即可；如果需要反复执行，至少做到
第 7 步；如果需要批量回归，做到第 9 步。

## 6. 创建项目和环境

### 6.1 创建项目

进入：

```text
项目管理 -> 新建项目
```

填写：

| 字段 | 如何填写 |
|---|---|
| 项目名称 | 例如：用户中心 |
| 项目标识 | 例如：user-center，可选 |
| Base URL | 项目说明用的基础地址，可选 |
| 项目描述 | 说明项目用途，可选 |

点击“确定”后，回到顶部项目选择器，选择刚创建的项目。

### 6.2 创建环境

进入：

```text
环境管理 -> 新建环境
```

填写：

| 字段 | 示例 |
|---|---|
| 环境名称 | 本地、测试环境、开发环境 |
| Base URL | `http://127.0.0.1:8001` 或目标服务地址 |
| 环境变量 | `{"token":"xxx","tenant_id":"demo"}` |
| 数据库配置 | 只有需要数据库断言时填写 |
| Cookie | 只有需要固定 Cookie 时填写 |

当前页面对环境 Base URL 做了 IP 地址校验，建议使用：

```text
http://127.0.0.1:端口
http://192.168.1.10:端口
https://10.0.0.5:端口
```

如果目标服务只能使用域名，快速调试时可以在 URL 中直接填写完整域名，
不要依赖环境 Base URL 的拼接。

环境变量必须是合法 JSON，例如：

```json
{
  "token": "test-token",
  "user_id": 10001,
  "tenant": "demo"
}
```

保存后，在顶部环境选择器选中该环境。

### 6.3 项目和环境的区别

- 项目：组织接口、用例、计划和报告。
- 环境：决定请求发到哪里，以及请求执行时使用哪些变量、Cookie 和数据库。
- 一个项目可以有多个环境，例如“本地”“测试”“预发布”。
- 执行前一定要确认选中的环境，尤其是删除、更新类接口。

## 7. 快速调试一个 API

进入：

```text
API 测试 -> 接口调试
```

### 7.1 最简单的 GET 请求

例如测试本地服务：

1. 方法选择 `GET`。
2. URL 输入 `http://127.0.0.1:8001/health/ready`。
3. 点击“发送请求”。
4. 在下方查看状态码、响应 Body、响应 Headers 和耗时。

发送成功后，页面会显示执行状态。返回 `200` 只能说明请求成功到达并返回
了 200，是否满足业务要求还需要配置断言。

### 7.2 请求配置

接口调试页顶部有方法、URL 和超时设置。下面有几个核心标签：

| 标签 | 用途 |
|---|---|
| Params | 查询参数，例如 `{"page":1,"size":20}` |
| Headers | 请求头，例如 `{"Content-Type":"application/json"}` |
| Body | JSON 请求体 |
| Auth | Bearer、OAuth2、API Key 或 Basic Auth |
| Tests | 状态码、JSON 路径、Header、响应时间等断言 |

Headers、Params 和 Body 使用 JSON。示例：

```json
{
  "Content-Type": "application/json",
  "X-Trace-Id": "demo-001"
}
```

```json
{
  "username": "alice",
  "password": "123456"
}
```

### 7.3 使用环境的 Base URL

选择环境后，可以在 URL 中填写相对路径：

```text
/api/users
```

后端会把它拼接到当前环境 Base URL 后面。例如：

```text
环境 Base URL：http://127.0.0.1:9000
请求 URL：/api/users
最终请求：http://127.0.0.1:9000/api/users
```

如果 URL 已经是 `http://` 或 `https://` 开头，平台会直接使用完整 URL，不再
拼接环境 Base URL。

### 7.4 配置断言

在 `Tests` 标签中添加断言。第一次建议只添加：

```text
类型：status_code
操作符：eq
期望值：200
```

常见断言：

- 状态码等于 200。
- JSON 路径 `$.data.id` 存在或等于某个值。
- Header 包含某个字段。
- 响应时间小于指定毫秒数。
- 响应符合 JSON Schema。

### 7.5 保存为接口用例

调试成功后，点击右上角“保存到接口列表”：

1. 填写标题。
2. 选择所属项目。
3. 设置模块/分组。
4. 确认方法、URL、Headers、Params、Body 和断言。
5. 保存。

保存后可以选择跳转到“接口定义”查看，也可以稍后从“用例管理”进入。

## 8. 保存 API 用例并执行

### 8.1 用例管理

进入：

```text
API 测试 -> 用例管理
```

这里适合维护可以反复执行的 API 用例。新建或编辑用例时主要有：

#### 基本信息

- 用例标题。
- 请求方法。
- 请求 URL。
- 模块/分组。
- 所属项目。
- Headers、Query Params、Body。

#### 高级配置

- 失败重试次数。
- 重试间隔。
- 前置 Python 脚本。
- 后置 Python 脚本。

#### 数据库断言

保存用例后再配置数据库断言。数据库断言只支持查询类 SQL，通常用于：

```sql
SELECT COUNT(*) FROM users WHERE username = '${username}'
```

配置数据库断言时需要选择环境，因为平台要使用该环境的数据库配置。

### 8.2 单独执行用例

在用例列表中找到目标用例，使用“运行”或进入接口调试页查看结果。执行时：

1. 确认用例所属项目。
2. 确认用例绑定的环境，或在执行页面选择环境。
3. 确认相对 URL 有可用的 Base URL。
4. 执行后到“任务中心”查看任务状态。

### 8.3 用例执行结果

常见状态：

| 状态 | 含义 |
|---|---|
| passed | 请求和断言都通过 |
| failed | 请求完成，但至少一个断言不通过 |
| error | 请求或执行过程发生错误 |
| skipped | 被跳过 |

不要只看 HTTP 状态码。接口返回 200 但业务字段错误时，仍可能因为断言失败
而显示 `failed`。

## 9. 测试计划

进入：

```text
API 测试 -> 测试计划
```

测试计划用于把多个 API 用例组织在一起，保存执行顺序、执行模式和筛选条件，
供后续批量或调度链路使用。

### 9.1 创建计划

点击“新建计划”，填写：

1. 所属项目。
2. 计划名称，例如“用户模块冒烟测试”。
3. 描述，可选。
4. 执行模式。
5. 标记筛选，可选，例如 `smoke`。

执行模式：

| 模式 | 适合场景 |
|---|---|
| sequential | 需要按顺序执行，适合冒烟和依赖链 |
| parallel | 用例之间相互独立，想缩短执行时间 |
| stress | 需要进行压力型批量执行，使用前先确认目标服务承受能力 |

创建后打开“详情”，点击“添加用例”，选择要加入计划的 API 用例。

### 9.2 当前页面的使用边界

当前前端的“测试计划”页面主要用于：

- 创建和编辑计划。
- 添加或移除 API 用例。
- 调整执行模式和标记筛选。
- 查看计划详情。

当前页面暂时没有直接的“执行计划”按钮。需要实际执行时，建议先在“用例
管理”中运行单个用例；需要定时或批量触发时，可在“定时任务”中直接选择
要执行的用例。计划链路的后端执行接口主要供 API/CI 调用，普通页面操作
不需要进入该接口。

## 10. 数据驱动

进入：

```text
API 测试 -> 数据驱动
```

数据驱动的前提是：**先有一个已经保存的 API 测试用例**。

### 10.1 创建数据集

1. 顶部“关联测试用例”选择一个 API 用例。
2. 点击“新建数据集”。
3. 填写数据集名称。
4. 选择 `CSV` 或 `JSON`。
5. 输入数据内容。
6. 查看页面中的变量解析和前 5 行预览。
7. 点击确定保存。

CSV 示例：

```csv
username,password
alice,1234
bob,5678
```

JSON 示例：

```json
[
  {
    "username": "alice",
    "password": "1234"
  },
  {
    "username": "bob",
    "password": "5678"
  }
]
```

第一行表头或 JSON 对象的键就是变量名。

### 10.2 在用例中引用变量

推荐使用标准语法：

```text
{{username}}
{{password}}
```

例如 Body：

```json
{
  "username": "{{username}}",
  "password": "{{password}}"
}
```

平台暂时兼容 `${username}` 语法，但新建内容建议统一使用 `{{username}}`。

### 10.3 执行数据驱动

1. 选择测试用例。
2. 确认列表中有启用的数据集。
3. 点击“数据驱动执行”。
4. 可选择某个数据集，也可以不选择，执行该用例下全部启用的数据集。
5. 选择执行环境。
6. 点击“执行”。
7. 查看每一行的输入数据、状态码、耗时和错误信息。

如果页面打不开或没有数据，先确认：

- 已经登录。
- 顶部或页面内已经选择测试用例。
- 该用例确实存在。
- 数据集已经保存且状态为“启用”。
- 数据内容是合法 CSV 或 JSON。

## 11. 接口导入和接口文档

### 11.1 OpenAPI/Swagger 导入

进入：

```text
API 测试 -> 接口导入
```

适合已有接口文档的情况：

1. 选择 OpenAPI/Swagger 导入。
2. 提供文档 URL 或 JSON 内容。

## 13. 性能测试

性能测试会真实访问目标服务，不要对生产系统随意执行。个人本地使用时，
建议先用低并发、短时间配置验证流程。

### 13.1 创建压测场景

进入：

```text
性能测试 -> 压测场景
```

点击“新建压测场景”，填写：

1. 场景名称。
2. 所属项目。
3. 关联 API 测试用例。
4. 压测模式。
5. 并发用户数。
6. 启动速率。
7. 持续时间。
8. 可选的 SLA 阈值。

| 模式 | 说明 |
|---|---|
| 稳定负载 | 维持相对稳定的用户数 |

| 自定义曲线 | 自己配置多个阶段 |

建议第一次使用：


- 错误率。

也可以进入：

```text

```

选择场景后查看实时指标。压测结束后，在“性能报告”查看 P50、P90、P95、
P99、RPS、错误率和 SLA 结果。

## 15. 变量、认证、脚本和 Cookie

### 15.1 变量优先级

执行时常用的变量来源从高到低为：

1. 当前请求临时变量。
2. 当前环境变量。
3. 全局或项目工作空间变量。

前置请求提取出的变量和前置脚本修改的变量，会继续传给后续请求。

引用示例：

```text
URL：https://example.test/users/{{user_id}}
Header：Bearer {{token}}
Body：{"name":"{{username}}"}
```

### 15.2 全局变量和工作空间变量

进入：

```text
变量管理 -> 新建变量
```

- **全局变量**：所有项目可使用。
- **工作空间变量**：必须绑定一个项目，只用于该项目。

变量类型可以选择字符串、数字、布尔或 JSON。敏感 Token 不建议作为普通
全局变量长期保存，使用前确认本机数据文件不会被提交到 Git。

### 15.3 认证

在接口调试页的 `Auth` 标签中选择：

- 无认证。
- Bearer Token。
- OAuth2。
- API Key，放在 Header 或 Query。
- Basic Auth。

认证信息只填写当前调试真正需要的内容。不要把真实生产 Token 粘贴到截图、
HAR 文件或代码仓库。

### 15.4 前置请求和变量提取

在“高级设置 -> 前置请求”中，可以配置登录请求、获取 Token 请求等依赖链：

1. 添加前置请求。
2. 填写名称、方法、URL、Headers 和 Body。
3. 添加变量提取规则。
4. 填写变量名、提取来源和表达式。
5. 主请求中使用提取出的变量。

常见提取来源：

- JSON 路径，例如 `$.data.token`。
- 正则表达式。
- 响应 Header。

### 15.5 前后置脚本

接口调试和用例高级配置支持 Python 前置/后置脚本。

前置脚本可访问 `variables` 和 `request`，例如：

```python
variables["trace_id"] = "demo-001"
```

后置脚本可访问 `response` 和 `variables`，例如：

```python
if response and response["status_code"] == 200:
    print("请求成功")
```

脚本运行在受限环境中，不能使用 `os`、`subprocess`、`sys`、`open`、`eval`
或 `exec`，也不能访问危险的双下划线属性。脚本报错时，优先检查变量名、
缩进和被禁止的调用。

### 15.6 Cookie

Cookie 有两种常用方式：

- 在环境管理中保存环境级 Cookie。
- 在接口调试页使用当前会话 Cookie，发送请求后自动捕获 `Set-Cookie`。

接口调试页的会话 Cookie 只在当前调试会话中保持，点击“清除全部”后不能恢复。

## 18. 常见问题

### 18.1 浏览器提示无法访问 `127.0.0.1:5173`

按顺序执行：

```powershell
Set-Location E:\xlwang\AIRETEST
.\scripts\status-local.ps1
```

如果前端不是运行状态，重新启动：

```powershell
.\scripts\start-local.ps1
```

仍然失败时检查：

```text
E:\xlwang\AIRETEST\.runtime\logs\frontend.out.log
E:\xlwang\AIRETEST\.runtime\logs\frontend.err.log
```

确认浏览器地址没有写成错误的端口。AIRETEST 前端是 `5173`，后端是 `8001`，
`8000` 是 C-Lodop 端口。

### 18.2 后端未就绪或页面登录后空白

检查：

```text
E:\xlwang\AIRETEST\.runtime\logs\backend.out.log
E:\xlwang\AIRETEST\.runtime\logs\backend.err.log
```

再确认：

1. `.env` 存在。
2. `DATABASE_URL=sqlite:///./airetest-lite.db`。
3. 磁盘和数据库文件可读写。
4. 没有其他程序占用 `8001`。

不要用 `AUTO_CREATE_SCHEMA=true` 绕过迁移。启动脚本会自动执行 Alembic
迁移，迁移失败时应先看后端错误日志。

### 18.3 登录失败或出现 401

401 通常表示登录状态失效或账号密码不正确：

1. 确认使用的是 `.runtime\initial-admin.txt` 中的初始账号，或你当前的密码。
2. 刷新页面后重新登录。
3. 如果页面仍显示旧状态，先退出登录，再重新打开
   `http://127.0.0.1:5173/login`。
4. 不要在 URL 中手工拼接 Token。

### 18.4 保存或执行时出现 422

422 通常表示提交的数据不符合表单要求。优先检查：

- Headers、Params、Body 是否是合法 JSON。
- 环境变量是否是合法 JSON。
- URL 是否为空。
- 环境 Base URL 是否为 `http://IP:端口` 或 `https://IP:端口`。
- 测试计划是否选择了所属项目。
- 数据驱动内容是否符合 CSV/JSON 格式。
- 数据库断言是否填写了 SQL 和比较值。

错误提示通常会直接显示具体字段。不要反复点击“保存”，先修正对应字段。

### 18.5 数据驱动页面打不开或没有数据

正确入口是：

```text
API 测试 -> 数据驱动
```

然后在页面顶部选择一个已保存的 API 测试用例。没有用例时，先去“接口调试”
发送请求并保存到接口列表。没有数据集时，点击“新建数据集”创建并保存。

### 18.6 测试套件打开失败或执行提示没有用例

正确入口是：

```text
UI 测试 -> 测试套件
```

测试套件必须先有 UI 用例：

1. 到“UI 用例管理”创建或保存一个 UI 用例。
2. 回到“测试套件”点击刷新。
3. 新建或编辑套件。
4. 在“选择用例”中选择 UI 用例。
5. 保存后再执行。

不要把 API 用例当作 UI 套件用例。

### 18.7 日志查询页没有数据

正确入口是：

```text
UI 测试 -> 日志查询
```

日志来自 UI 用例或 UI 套件的执行。刚打开页面没有数据时：

1. 确认已经执行过至少一次 UI 用例或 UI 套件。
2. 点击“刷新”。
3. 清除项目、时间和日志级别筛选。
4. 再查看是否有 `INFO`、`WARN` 或 `ERROR` 日志。

### 18.8 相对 URL 请求失败

相对 URL 必须有选中的环境和可用的 Base URL。检查：

```text
顶部环境选择器是否有值
环境 Base URL 是否正确
请求是否写成 /api/users
```

也可以直接在 URL 中填写完整的 `http://` 或 `https://` 地址绕过 Base URL
拼接。

### 18.9 UI 测试无法启动浏览器

执行：

```powershell
python -m playwright install chromium
```

如果仍然失败，查看任务中心详情和 `.runtime\logs`，确认不是目标页面地址、
元素选择器或权限问题。

### 18.10 性能测试一直运行或没有实时数据

检查：

1. 压测场景是否关联 API 用例。
2. 并发用户数和持续时间是否大于 0。
3. 目标接口是否可访问。
4. 任务中心是否显示任务正在运行。
5. 性能页面是否仍在轮询。
6. 后端是否被手工关闭。

本地线程模式下，后端进程退出会中断任务。先执行 `status-local.ps1`，再
查看后端日志。

### 18.11 页面菜单里找不到功能

当前默认是 `lite` 菜单。可以：

1. 先确认功能是否在 API 测试、UI 测试或性能测试分组下。
2. 点击对应分组展开子菜单。
3. 需要完整菜单时执行：

```powershell
.\scripts\start-local.ps1 -FrontendMode full
```

### 18.12 浏览器显示旧页面

使用：

```text
Ctrl + Shift + R
```

强制刷新。若仍有问题，退出登录、关闭浏览器标签页后重新访问
`http://127.0.0.1:5173`。

## 20. 本地文件和功能边界

### 20.1 重要本地目录

| 路径 | 内容 |
|---|---|
| `airetest-lite.db` | SQLite 业务数据库 |
| `.uploads` | 测试产物，例如截图、Trace、报告 |
| `.runtime` | PID、日志、初始管理员信息 |
| `.runtime\logs` | 前后端启动和运行日志 |
| `.backups` | 本地备份 |
| `.env` | 本地配置和密钥，不应提交到 Git |

这些文件是本地运行数据，不要随意删除，也不要提交到公开仓库。

### 20.2 轻量单机模式的限制

轻量模式适合你本人在一台电脑上使用，不是高可用或分布式部署模式：

- 任务在当前后端进程内的本地线程中执行。
- 重启或强制结束后端时，运行中的任务可能中断。
- 没有独立 Worker 接管、租约、重投递和多机恢复。
- 长时间 UI、性能或脚本任务执行时，不要随意停止平台。
- 需要 Oracle、Redis、Celery 或分布式 Worker 时，才考虑完整模式和
  `ORACLE部署与迁移.md`。
